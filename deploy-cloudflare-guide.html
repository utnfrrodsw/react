Deploy de Frontend en Cloudflare Pages

Guia completa paso a paso — desde el repo local hasta produccion con la CDN mas rapida del mundo.

✅ Prerequisitos

Antes de empezar, asegurate de tener:

  • Una cuenta en Cloudflare (el plan Free incluye Pages con builds ilimitados)
  • Node.js v18+ instalado (node -v)
  • Un repositorio en GitHub, GitLab o Bitbucket
  • El CLI de Wrangler instalado:
npm i -g wrangler
Tip: Wrangler es el CLI oficial de Cloudflare. Reemplaza al viejo wrangler y sirve tanto para Pages como para Workers.

📁 Estructura del proyecto

Cloudflare Pages funciona con cualquier framework estatico. Asegurate de que tu proyecto tenga esta estructura minima:

mi-frontend/ package.json ← tiene que tener el script "build" vite.config.js ← o webpack, o next.config.js .env ← env vars locales (NO commitear) .env.example ← template para el equipo src/ components/ pages/ main.jsx App.jsx public/ favicon.ico

Verifica que tu package.json tenga el script de build:

{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
Importante: El directorio de build por defecto es dist (Vite) o build (CRA). Cloudflare Pages lo detecta, pero si usas otro output dir, configuralo en el paso 6.

🔧 Variables de entorno

Las env vars son la parte mas critica. Cloudflare Pages las maneja en dos niveles:

a) Localmente (en tu maquina)

Crea un archivo .env en la raiz del proyecto:

# .env (NO commitear a git)

# API
VITE_API_URL=https://api.miapp.com/v1
VITE_API_KEY=sk_live_abc123def456

# Auth
VITE_AUTH0_DOMAIN=miauth0.auth0.com
VITE_AUTH0_CLIENT_ID=abc123xyz
VITE_AUTH0_REDIRECT_URI=http://localhost:5173/callback

# Features
VITE_ENABLE_ANALYTICS=true
VITE_ENVIRONMENT=development
Nunca commitees archivos .env con secrets reales. Agrega .env a tu .gitignore. Solo commitea .env.example o .env.local como referencia. 
# .gitignore
.env
.env.local
.env.*.local

Crea el template para tu equipo:

# .env.example (este SI se commitea)

VITE_API_URL=
VITE_API_KEY=
VITE_AUTH0_DOMAIN=
VITE_AUTH0_CLIENT_ID=
VITE_AUTH0_REDIRECT_URI=
VITE_ENABLE_ANALYTICS=false
VITE_ENVIRONMENT=development

b) Prefijo correcto segun el bundler

Bundler / FrameworkPrefijoEjemplo
ViteVITE_VITE_API_URL
Next.js (estatico)NEXT_PUBLIC_NEXT_PUBLIC_API_URL
Create React AppREACT_APP_REACT_APP_API_URL
Clave: Solo las env vars con el prefijo correcto se exponen al cliente. Las que no lo tienen quedan embebidas en el build y no estan disponibles en import.meta.env.

📦 Wrangler CLI — Autenticacion y proyecto

Antes de deployar, autenticate con Cloudflare y crea el proyecto:

Login con Cloudflare

# Abrira el browser para autenticarte
wrangler login

Crear proyecto de Pages

# Crear un nuevo proyecto (solo la primera vez)
wrangler pages project create mi-frontend

# Te va a preguntar:
# - Nombre del proyecto
# - Branch principal (main)
# - Build command (detecta automaticamente)
# - Output directory (detecta automaticamente)

Deploy directo desde CLI (alternativa)

# Build y deploy en un solo comando
wrangler pages deploy dist --project-name=mi-frontend

# O con deploy preview
wrangler pages deploy dist --project-name=mi-frontend --branch=feature-nueva
Ventaja: wrangler pages deploy sube el build estatico directamente a la CDN sin necesidad de GitHub. Ideal para CI/CD custom o deploy rapido.

🔗 Conectar el repositorio

Al conectar el repo, Cloudflare Pages crea un proyecto vinculado. Cada push a main (o la branch que configures) automaticamente despliega a produccion.

Desde la web (recomendado para el primer deploy)

  1. Anda a dash.cloudflare.com
  2. Selecciona tu cuenta
  3. Anda a Workers & Pages → Create → Pages → Connect to Git
  4. Selecciona tu proveedor (GitHub/GitLab/Bitbucket)
  5. Autoriza el acceso y selecciona el repositorio
  6. Configura el Project name
  7. Configura Production branch (ej: main)
  8. En Build settings, completa los campos (ver paso 6)
  9. Hace click en "Save and Deploy"

Flujo de un push:

git push origin main
  ↓
Cloudflare detecta el push
  ↓
Instala dependencias (npm install)
  ↓
Ejecuta el build (npm run build)
  ↓
Sube archivos a la CDN global de Cloudflare
  ↓
Asigna una URL: mi-frontend.pages.dev
  ↓
Si tiene dominio custom → actualiza produccion
Git integration: Cloudflare Pages crea checks de status en GitHub que muestran el estado del deploy en cada PR.

Monorepo: sub-directorio del frontend

Si tu repo tiene backend + frontend:

# En Cloudflare Dashboard: Settings → Builds & deploy
# Pone la carpeta del frontend:
Root directory: packages/frontend

# Build command:
cd ../.. && npm install && cd packages/frontend && npm run build

# Build output directory:

Archivo wrangler.toml (opcional)

Para configuraciones avanzadas, puedes crear un wrangler.toml en la raiz del proyecto:

# wrangler.toml
name = "mi-frontend"
compatibility_date = "2024-01-01"

# Configuracion de Pages
[site]
bucket = "./dist"

# Headers personalizados
[[headers]]
  pattern = "/*"
  [headers.values]
    X-Frame-Options = "DENY"
    X-Content-Type-Options = "nosniff"

# Redirects
[[redirects]]
  from = "/old-page"
  to = "/new-page"
  status = 301

# Reglas SPA (equivalente a vercel.json rewrites)
[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
SPA routing: Si usas React Router o similar, la regla de redirect /* → /index.html es obligatoria. Sin ella, recargar en una ruta que no sea / devuelve 404.

📋 Cargar env vars en Cloudflare Pages

Cloudflare Pages tiene 3 ambitos para env vars. Esto es fundamental para no romper nada:

AmbitoCuando se usanPara que
Production Deploy a main URLs de API reales, secrets de prod
Preview PRs y branches no-main API staging, datos de test
Deploy hook Webhook manual Trigger de deploy via URL o API

Desde el dashboard

  1. Workers & Pages → mi-frontend → Settings → Environment variables
  2. Selecciona el ambito: Production o Preview
  3. Escribi el nombre (VITE_API_URL)
  4. Escribi el valor
  5. Hace click en "Encrypt" si es un secret
  6. Hace click en "Add"
  7. Repite para cada variable

Desde Wrangler CLI

# Agregar una variable de produccion
wrangler pages secret put VITE_API_URL --project-name=mi-frontend
# → Ingresar el valor en el prompt

# Agregar una variable de produccion (no secret)
wrangler pages secret put VITE_API_URL --project-name=mi-frontend --production

# Agregar una variable de preview
wrangler pages secret put VITE_API_URL --project-name=mi-frontend --preview

# Listar secrets
wrangler pages secret list --project-name=mi-frontend

# Eliminar un secret
wrangler pages secret delete VITE_API_URL --project-name=mi-frontend

Ejemplo practico para un proyecto React

# Production (produccion real)
wrangler pages secret put VITE_API_URL --project-name=mi-frontend
# → https://api.miapp.com/v1

wrangler pages secret put VITE_API_KEY --project-name=mi-frontend
# → sk_live_abc123def456

# Preview (staging para PRs)
wrangler pages secret put VITE_API_URL --project-name=mi-frontend --preview
# → https://api-staging.miapp.com/v1

wrangler pages secret put VITE_API_KEY --project-name=mi-frontend --preview
# → sk_test_xyz789ghi012
Cuidado: Cuando agregas o cambias una env var, se aplica en el proximo deploy. Si necesitas que aplique inmediatamente, hace un redeploy manual. 
# Forzar redeploy para que tomen las nuevas env vars
wrangler pages deploy dist --project-name=mi-frontend

# O desde el dashboard: Deployments → "..." → Retry deployment
Secrets enmascarados: Cloudflare enmascara automaticamente los secrets en el dashboard. Una vez creados, no se pueden volver a leer. Anotalos en un gestor de secrets.

🎨 Deploy previews y branches

Cada branch o PR crea un deploy automatico con una URL unica:

# Cada PR genera una URL como:
https://mi-frontend.branch-name.pages.dev

Esto permite:

  • Review de UI antes de mergear — el equipo puede ver los cambios en vivo
  • Testing automatizado — podes correr E2E contra la URL de preview
  • Separacion de env vars — los previews usan las vars del ambito "Preview", no las de prod

Configurar branch deployments

En Settings → Builds & deploy → Branch deployments:

BranchComportamiento
mainDeploy a produccion automatico
developDeploy preview
Otras branchesDeploy preview automatico

🌐 Dominio personalizado

Para usar tu propio dominio en vez de mi-frontend.pages.dev:

Opcion A: Tu dominio ya usa Cloudflare DNS

  1. Workers & Pages → mi-frontend → Custom domains
  2. Hace click en "Set up a custom domain"
  3. Escribe tu dominio (ej: miapp.com)
  4. Cloudflare configura los DNS records automaticamente
  5. SSL se genera y renueva automaticamente

Opcion B: Tu dominio usa otro DNS provider

Si el DNS lo maneja otro proveedor, configura estos registros:

# Tipo A (para apex domain)
@     A     192.0.2.1

# Tipo CNAME (para subdomain)
www   CNAME mi-frontend.pages.dev
Nota: Si tu dominio esta en otro proveedor, es mas facil migrar el DNS a Cloudflare primero. Asi aprovechas la CDN, el WAF, y el SSL automatico sin config extra.
SSL automatico: Cloudflare genera y renueva certificados SSL/TLS automaticamente. En Pages, el certificado lo gestiona Cloudflare, no necesitas configurar nada.
Propagation: Los cambios de DNS pueden tardar hasta 48hs, pero normalmente son 5-30 minutos. Cloudflare suele ser mas rapido que otros proveedores.

⚡ Cloudflare Functions (Opcional)

Cloudflare Pages soporta Functions (antes conocido como Pages Functions). Podes usarlas para logica backend sin un servidor dedicado.

Estructura

mi-frontend/ functions/ ← Cloudflare detecta esta carpeta api/ hello.js ← → /api/hello users/[id].js ← → /api/users/:id _middleware.js ← middleware global (auth, CORS, etc.) src/ dist/ ← output del build

Ejemplo de function

// functions/api/hello.js
export async function onRequest(context) {
  return new Response(JSON.stringify({ message: "Hello from Cloudflare!" }), {
    headers: { "Content-Type": "application/json" },
  });
}

Ejemplo con parametros

// functions/api/users/[id].js
export async function onRequest(context) {
  const { id } = context.params;
  // Acceder a secrets: context.env.API_KEY
  // Acceder a KV: context.env.MY_KV.get(`user:${id}`)
  return new Response(JSON.stringify({ userId: id }), {
    headers: { "Content-Type": "application/json" },
  });
}

Middleware (CORS, Auth)

// functions/_middleware.js
export async function onRequest(context) {
  // CORS headers
  const response = await context.next();
  response.headers.set("Access-Control-Allow-Origin", "*");
  response.headers.set("Access-Control-Allow-Methods", "GET, POST, OPTIONS");
  return response;
}
Ventaja: Las Functions se ejecutan en la edge (el punto mas cercano al usuario), lo que las hace extremadamente rapidas. No necesitas configurar un servidor separado.

✅ Post-deploy: verificacion

Despues del primer deploy, verificaste todo:

  • La app carga correctamente en la URL de Cloudflare Pages
  • Las llamadas a la API funcionan (no 404 ni CORS errors)
  • Las variables de entorno estan disponibles (console.log(import.meta.env.VITE_API_URL))
  • El routing funciona (recarga en /otra-ruta no da 404)
  • Los assets statics cargan (favicon, imagenes, fonts)
  • El HTTPS funciona correctamente
  • Los headers de cache estan bien (Network tab en DevTools)
  • Un deploy preview funciona en una branch de test

Debugging rapido

# Ver logs de Functions en tiempo real
wrangler pages deployment tail --project-name=mi-frontend

# Abrir el dashboard del proyecto
wrangler pages project list
Real-time logs: Podes ver los logs de Functions en tiempo real desde el dashboard de Cloudflare en Workers & Pages → mi-frontend → Logs.

🚀 CI/CD avanzado

Para equipos y proyectos en produccion, estas configuraciones son esenciales:

Proteger produccion

Workers & Pages → mi-frontend → Settings → Builds & deploy:

  • Branch protection rules — requiere approval antes de deployar a produccion
  • Required reviewers — para dominios criticos
  • Build caches — habilitar caches para builds mas rapidos

Deploy hooks

# Crear un deploy hook en Dashboard:
# Settings → Builds & deploy → Deploy hooks

# Ejemplo: Deploy manual via curl
curl -X POST https://api.cloudflare.com/client/v4/pages/webhooks/deploy/hooks/HOOK_ID \
  -H "Authorization: Bearer YOUR_API_TOKEN"

Cache de builds

# Cloudflare Pages cachea automaticamente las dependencias
# Si el build es lento, verifica en:
# Settings → Builds & deploy → Build caches

# Para invalidar el cache, agrega esto al build command:
npm ci --cache /tmp/.npm && npm run build

Deploy con GitHub Actions (alternativa)

# .github/workflows/deploy.yml
name: Deploy to Cloudflare Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run build
      - name: Deploy to Cloudflare Pages
        uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          accountId: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          command: pages deploy dist --project-name=mi-frontend
API Token: Crea un API token en Cloudflare Dashboard → My Profile → API Tokens → Create Token. Usa el template "Edit Cloudflare Workers".

Analytics (Web Vitals)

# Cloudflare Web Analytics (gratis)
# 1. Dashboard → Analytics & Logs → Web Analytics
# 2. Copia el script tag

🔥 Troubleshooting

Error: "Build failed"

# 1. Corri el build localmente para ver el error real
npm run build

# 2. Revisa los Build Logs en el dashboard:
#    Workers & Pages → mi-frontend → Deployments → [tu deploy] → Build logs

Error: "404 en recarga de pagina"

Falta la regla de redirect SPA. Agrega a wrangler.toml:

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200

O configuralo desde el dashboard en Routes.

Error: "Variables de entorno undefined"

# 1. Verifica el prefijo (VITE_, NEXT_PUBLIC_, REACT_APP_)
# 2. Verifica el ambito (Production vs Preview)
# 3. Hace redeploy despues de agregar las env vars
wrangler pages deploy dist --project-name=mi-frontend

# 4. Verifica que esten configuradas:
wrangler pages secret list --project-name=mi-frontend

Error: "CORS en produccion"

# Agrega CORS en una Function middleware:
// functions/_middleware.js
export async function onRequest(context) {
  const response = await context.next();
  response.headers.set("Access-Control-Allow-Origin", "https://miapp.com");
  response.headers.set("Access-Control-Allow-Methods", "GET, POST, OPTIONS");
  return response;
}

# O en tu backend (Express):
const cors = require('cors');
app.use(cors({
  origin: [
    'https://miapp.com',
    'https://www.miapp.com',
    'https://mi-frontend.pages.dev'  // para previews
  ]
}));

Build lento

# 1. Cloudflare Pages ya cachea dependencias automaticamente
# 2. Si usas monorepo, configura solo el package del frontend:
#    Build command: cd packages/frontend && npm install && npm run build

# 3. Usa pnpm en vez de npm (mas rapido):
#    Build command: npm i -g pnpm && pnpm install --frozen-lockfile && pnpm build

Como hacer rollback

# Desde el dashboard:
# Deployments → seleccionar un deploy anterior → "..." → Rollback to this deployment

# Desde la CLI (deploy un commit anterior):
wrangler pages deploy dist --project-name=mi-frontend

Error: "Functions not found" o "404 en /api"

# 1. Verifica que la carpeta se llame "functions" (en minuscula)
# 2. Verifica la estructura de archivos:
#    functions/api/hello.js → /api/hello
#    functions/api/users/[id].js → /api/users/:id

# 3. Verifica que el nombre del archivo coincida con la ruta
# 4. Haz redeploy despues de agregar Functions

📋 Checklist final

  • Repo conectado a Cloudflare Pages
  • Framework detectado correctamente
  • Build command y output dir configurados
  • Env vars cargadas en Production
  • Env vars cargadas en Preview
  • .env esta en .gitignore
  • .env.example commiteado con los nombres de las vars
  • SPA routing configurado (redirect /* a /index.html)
  • Primer deploy funciona correctamente
  • Deploy preview funciona en una branch
  • Dominio personalizado configurado (si aplica)
  • SSL activo y funcionando
  • Functions configuradas (si aplica)
  • Rollback probado
