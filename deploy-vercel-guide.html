Deploy de Frontend en Vercel

Guia completa paso a paso — desde el repo local hasta produccion.

✅ Prerequisitos

Antes de empezar, asegurate de tener:

  • Una cuenta en vercel.com (el plan Hobby es gratis y alcanza para proyectos personales)
  • Node.js v18+ instalado (node -v)
  • Un repositorio en GitHub, GitLab o Bitbucket
  • El CLI de Vercel instalado (opcional pero recomendado):
npm i -g vercel
Tip: Podes hacer todo desde la CLI o desde la interfaz web. Esta guia cubre ambos flujos.

📁 Estructura del proyecto

Vercel detecta automaticamente frameworks como React, Next.js, Vue, Angular, etc. Asegurate de que tu proyecto tenga esta estructura minima:

mi-frontend/ package.json ← tiene que tener el script "build" vite.config.js ← o webpack, o next.config.js .env ← env vars locales (NO commitear) .env.example ← template para el equipo src/ components/ pages/ main.jsx App.jsx public/ favicon.ico

Verifica que tu package.json tenga el script de build:

{
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}
Importante: El directorio de build por defecto es dist (Vite) o build (CRA). Vercel lo detecta solo, pero si usas otro output dir, configuralo en el paso 6.

🔧 Variables de entorno

Las env vars son la parte mas critica. Hay que manejarlas en dos niveles:

a) Localmente (en tu maquina)

Crea un archivo .env en la raiz del proyecto:

# .env (NO commitear a git)

# API
VITE_API_URL=https://api.miapp.com/v1
VITE_API_KEY=sk_live_abc123def456

# Auth
VITE_AUTH0_DOMAIN=miauth0.auth0.com
VITE_AUTH0_CLIENT_ID=abc123xyz
VITE_AUTH0_REDIRECT_URI=http://localhost:5173/callback

# Features
VITE_ENABLE_ANALYTICS=true
VITE_ENVIRONMENT=development
Nunca commitees archivos .env con secrets reales. Agrega .env a tu .gitignore. Solo commitea .env.example o .env.local como referencia. 
# .gitignore
.env
.env.local
.env.*.local

Crea el template para tu equipo:

# .env.example (este SI se commitea)

VITE_API_URL=
VITE_API_KEY=
VITE_AUTH0_DOMAIN=
VITE_AUTH0_CLIENT_ID=
VITE_AUTH0_REDIRECT_URI=
VITE_ENABLE_ANALYTICS=false
VITE_ENVIRONMENT=development

b) Prefijo correcto segun el bundler

Bundler / FrameworkPrefijoEjemplo
ViteVITE_VITE_API_URL
Next.jsNEXT_PUBLIC_NEXT_PUBLIC_API_URL
Create React AppREACT_APP_REACT_APP_API_URL
Webpack manualNEXT_PUBLIC_NEXT_PUBLIC_API_URL
Clave: Solo las env vars con el prefijo correcto se exponen al cliente. Las que no lo tienen quedan en el servidor y no estan disponibles en import.meta.env.

c) Variables solo para Build-time

Son variables que se reemplazan during el build y no se pueden cambiar sin redeploy:

# Se usan en build, NO disponibles en runtime
VITE_API_URL=https://api.miapp.com/v1   # fija para el build

Si necesitas variables en runtime (cambiables sin redeploy), usa el SDK de Vercel:

# Instalar el SDK de Vercel Edge Config
npm i @vercel/edge-config

# O usar fetch para Vercel env API (solo en serverless functions)

⚙️ Configuracion de Vercel

Tenes dos formas de configurar el deploy:

Opcion A: Desde la web (recomendado para el primer deploy)

  1. Anda a vercel.com/new
  2. Hace click en "Import Git Repository"
  3. Selecciona tu proveedor (GitHub/GitLab/Bitbucket)
  4. Autoriza el acceso y selecciona el repositorio
  5. Vercel detecta automaticamente el framework y configura los settings
  6. Antes de hacer deploy, expandi "Environment Variables" y carga tus variables
  7. Hace click en "Deploy"

Opcion B: Desde la CLI

# Desde la raiz de tu proyecto
vercel

# Te va a guiar por:
# 1. Login (abre browser)
# 2. Seleccionar o crear proyecto
# 3. Directorio del codigo (./ por defecto)
# 4. Confirmar build command y output dir
# 5. Deploy!

🔗 Conectar el repositorio

Al conectar el repo, Vercel crea un proyecto vinculado. Cada push a main (o la branch que configures) automaticamente despliega a produccion.

Flujo de un push:

git push origin main
  ↓
Vercel detecta el push
  ↓
Instala dependencias (npm install / pnpm install)
  ↓
Ejecuta el build (npm run build)
  ↓
Sube los archivos a la CDN global
  ↓
Asigna una URL: mi-frontend-abc123.vercel.app
  ↓
Si esta vinculado a dominio custom → actualiza produccion
Git integration: Vercel crea un bot en GitHub que comenta en cada PR con la URL del deploy preview.

Monorepo: sub-directorio del frontend

Si tu repo tiene backend + frontend:

# En Vercel Dashboard: Settings → General → Root Directory
# Pone la carpeta del frontend:
Root Directory: packages/frontend

# O en vercel.json:
{
  "buildCommand": "cd packages/frontend && npm run build",
  "outputDirectory": "packages/frontend/dist"
}

Archivo vercel.json (opcional)

Para configuraciones avanzadas, crea vercel.json en la raiz:

{
  "buildCommand": "npm run build",
  "outputDirectory": "dist",
  "framework": "vite",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ],
  "headers": [
    {
      "source": "/assets/(.*)",
      "headers": [
        { "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }
      ]
    }
  ],
  "redirects": [
    { "source": "/old-page", "destination": "/new-page", "permanent": true }
  ]
}
SPA routing: Si usas React Router o similar, la regla de rewrites es obligatoria. Sin ella, recargar en una ruta que no sea / devuelve 404.

📋 Cargar env vars en Vercel

Vercel tiene 3 ambitos para env vars. Esto es fundamental para no romper nada:

AmbitoCuando se usanPara que
Production Deploy a main URLs de API reales, secrets de prod
Preview PRs y branches no-main API staging, datos de test
Development vercel dev local Entorno local con vars reales sin exponerlas

Desde el dashboard

  1. Proyecto → Settings → Environment Variables
  2. Escribi el nombre (VITE_API_URL)
  3. Escribi el valor
  4. Selecciona el ambito: Production / Preview / Development
  5. Hace click en "Add"
  6. Repite para cada variable

Desde la CLI

# Agregar una variable de produccion
vercel env add VITE_API_URL production

# Agregar una variable de preview (para PRs)
vercel env add VITE_API_URL preview

# Agregar una variable de desarrollo (vercel dev)
vercel env add VITE_API_URL development

# Listar todas las variables
vercel env ls

# Eliminar una variable
vercel env rm VITE_API_URL production

Ejemplo practico para un proyecto React

# Production (produccion real)
vercel env add VITE_API_URL production
# → https://api.miapp.com/v1

vercel env add VITE_API_KEY production
# → sk_live_abc123def456

# Preview (staging para PRs)
vercel env add VITE_API_URL preview
# → https://api-staging.miapp.com/v1

vercel env add VITE_API_KEY preview
# → sk_test_xyz789ghi012

# Development (local con vercel dev)
vercel env add VITE_API_URL development
# → http://localhost:3001/api

vercel env add VITE_API_KEY development
# → sk_test_local_key
Cuidado: Cuando agregas o cambias una env var, NO se redeploya automaticamente. Tenes que hacer un redeploy manual o un push para que los nuevos valores tomen efecto. 
# Forzar redeploy para que tomen las nuevas env vars
vercel --force

# O desde el dashboard: Deployments → "..." → Redeploy

Env vars sensibles

Vercel enmascara automaticamente las env vars en el dashboard (solo se muestran una vez al crearlas). Ademas, si alguien intenta loguearlas, el SDK las detecta y las oculta.

Reglas de seguridad:

  • Nunca uses VITE_SECRET_KEY — si tiene el prefijo VITE_, Vite la expone al cliente. Usa SECRET_KEY (sin prefijo) para server-side.
  • Los secrets de OAuth, JWT signing, etc. nunca deberian estar en variables con prefijo VITE_.
  • Para secrets que solo el backend necesita, usa Serverless Functions o Edge Functions.

🎨 Deploy previews y branches

Cada branch o PR crea un deploy automatico con una URL unica:

# Cada PR genera una URL como:
https://mi-frontend-git-feature-nueva-funcionalidad-miuser.vercel.app

Esto permite:

  • Review de UI antes de mergear — el equipo puede ver los cambios en vivo
  • Testing automatizado — podes correr E2E contra la URL de preview
  • Separacion de env vars — los previews usan las vars del ambito "Preview", no las de prod

Configurar branch deployments

En Settings → Git:

BranchComportamiento
mainDeploy a produccion automatico
developDeploy preview (se puede configurar como Production tambien)
Otras branchesDeploy preview automatico

🌐 Dominio personalizado

Para usar tu propio dominio en vez de mi-app.vercel.app:

  1. Settings → Domains
  2. Agrega tu dominio (ej: miapp.com)
  3. Vercel te da los DNS records que necesitas configurar:
# Opcion recomendada: DNS en tu proveedor (ej: Namecheap, Cloudflare)

# Tipo A (para apex domain)
@     A     76.76.21.21

# Tipo CNAME (para subdomain)
www   CNAME cname.vercel-dns.com
SSL automatico: Vercel genera y renueva certificados SSL automaticamente. No tenes que configurar nada.
Propagation: Los cambios de DNS pueden tardar hasta 48hs, pero normalmente son 5-30 minutos.

✅ Post-deploy: verificacion

Despues del primer deploy, verificaste todo:

  • La app carga correctamente en la URL de Vercel
  • Las llamadas a la API funcionan (no 404 ni CORS errors)
  • Las variables de entorno estan disponibles (console.log(import.meta.env.VITE_API_URL))
  • El routing funciona (recarga en /otra-ruta no da 404)
  • Los assets statics cargan (favicon, imagenes, fonts)
  • El HTTPS funciona correctamente
  • Los headers de cache estan bien (Network tab en DevTools)
  • Un deploy preview funciona en una branch de test

Debugging rapido

# Ver logs en tiempo real desde la CLI
vercel logs mi-app.vercel.app

# Ver logs del build
vercel logs mi-app.vercel.app --build

# Abrir el deploy en el browser
vercel open
Function logs: Si usas Serverless Functions, podes ver los logs en Runtime Logs del dashboard.

🔥 Troubleshooting

Error: "Build failed"

# 1. Corri el build localmente para ver el error real
npm run build

# 2. Revisa los Build Logs en el dashboard:
#    Deployments → [tu deploy] → Build Logs

Error: "404 en recarga de pagina"

Falta la regla de rewrite. Agrega a vercel.json:

{
  "rewrites": [
    { "source": "/((?!assets/).*)", "destination": "/index.html" }
  ]
}

Error: "Variables de entorno undefined"

# 1. Verifica el prefijo (VITE_, NEXT_PUBLIC_, REACT_APP_)
# 2. Verifica el ambito (Production vs Preview vs Development)
# 3. Hace redeploy despues de agregar las env vars
vercel --force

# 4. Verifica que esten en el scope correcto:
vercel env ls

Error: "CORS en produccion"

# El backend tiene que aceptar el dominio de Vercel
# Ejemplo en Express:
const cors = require('cors');

app.use(cors({
  origin: [
    'https://miapp.com',
    'https://www.miapp.com',
    'https://mi-frontend-abc123.vercel.app'  // para previews
  ]
}));

Build lento

# 1. Cache de dependencias: Vercel ya lo hace automaticamente
# 2. Si usas monorepo, configura solo el package del frontend:
#    Build → Install Command: cd packages/frontend && npm install

# 3. Usa pnpm en vez de npm (mas rapido):
#    Install Command: pnpm install --frozen-lockfile
#    Build Command: pnpm build

Como hacer rollback

# Desde el dashboard:
# Deployments → seleccionar un deploy anterior → "..." → Promote to Production

# Desde la CLI:
vercel rollback
